MODULE 1: DETECTION ENGINE ✅
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✓ Processes video clips from dataset
✓ Uses folder names (0_* through 6_*) for classification
✓ Maps to 7 behavior classes
✓ All required fields in output (clip_id, timestamp, behavior_class, policy_rule_ref, description, zone)
✓ Single video processing
✓ Batch processing support
✓ Train/test dataset support

HOW VIDEOS ARE DETECTED:
────────────────────────
1. Video file path is analyzed to extract folder name (e.g., 0_safe_walkway_violation)
2. Folder name is matched against BEHAVIOR_CLASS_MAPPING in app/config.py
3. OpenCV (cv2.VideoCapture) opens the video file
4. Video properties extracted: FPS, frame count, duration
5. One detection generated per video with:
   - clip_id: Video filename (without extension)
   - timestamp: Mid-video frame time (frame_count/2 ÷ FPS)
   - behavior_class: Mapped from folder name (e.g., "Carrying Overload with Forklift")
   - policy_rule_ref: Policy code (e.g., "POL-FOR-01")
   - description: Behavior description from config
   - zone: Location (default "Factory Floor A", customizable)
   - confidence: 1.0 (based on ground-truth folder structure)

Implementation Details (detection_engine.py):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
- extract_behavior_class_from_filename(): Parses path for folder match
- extract_policy_ref_from_folder(): Gets policy reference code
- process_video(): Single video processing via OpenCV
  * Opens video with cv2.VideoCapture()
  * Reads FPS and frame count
  * Calculates mid-point timestamp
  * Creates DetectionResult with all required fields
- process_batch_videos(): Multiple videos from directory
  * Recursively finds all .mp4 files
  * Processes each with process_video()
- process_dataset(): Entire train/test split with optional filtering
  * Supports behavior class filtering
  * Processes organized by class folder


MODULE 2: SEVERITY CATEGORIZATION MATRIX ✅
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✓ 4 severity tiers: LOW, MEDIUM, HIGH, CRITICAL
✓ Safe Walkway Violation → MEDIUM (based on "WARNING" in policy)
✓ Unauthorized Intervention → HIGH (based on "CRITICAL" in policy)
✓ Opened Panel Cover → LOW (based on "WARNING" in policy)
✓ Carrying Overload with Forklift → CRITICAL (based on "CRITICAL" in policy)
✓ Safe behaviors → NONE (no escalation)

Implementation (severity_classifier.py):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
- classify_severity(behavior_class): Maps behavior to severity
- should_trigger_real_time_alert(): Returns TRUE for HIGH/CRITICAL
- should_log_to_database(): Returns TRUE for all non-NONE
- get_escalation_action(): Returns action text per severity
- get_alert_color(): Returns color code (#FF0000 for red CRITICAL, etc.)

Severity Levels:
NONE (0)     → No action, safe behavior
LOW (1)      → Database log only
MEDIUM (2)   → Database log only
HIGH (3)     → Database log + real-time alert
CRITICAL (4) → Database log + real-time alert + supervisor notification


MODULE 3: ESCALATION PIPELINE ✅
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✓ LOW/MEDIUM → Log to database only (no alert)
✓ HIGH/CRITICAL → Log to database + trigger real-time WebSocket alert
✓ WebSocket support for real-time alerts
✓ Multiple client connections supported
✓ Alert payload includes severity, behavior, zone, timestamp

Routing Logic (escalation_pipeline.py):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. Event created with detected behavior
2. Severity classifier assigns tier
3. If HIGH/CRITICAL: WebSocket alert broadcast to all connected clients
4. All violations (except NONE) saved to database
5. Reports generated for all violations

Alert WebSocket Format:
{
  "event_id": "evt_20240115_143052_abc123",
  "severity": "CRITICAL",
  "behavior_class": "Carrying Overload with Forklift",
  "zone": "Factory Floor A",
  "timestamp": "2024-01-15T14:30:52Z",
  "message": "CRITICAL ALERT: Carrying Overload with Forklift detected"
}
